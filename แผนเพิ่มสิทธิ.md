🛠️ 2. ขั้นตอนการปรับปรุงระบบ (Step-by-Step Implementation Roadmap)
เพื่อไม่ให้โค้ดเก่าพัง เราจะใช้วิธี "รักษาเงื่อนไขเดิมของ Admin ไว้ แล้วค่อยๆ เปลี่ยนแปลงในจุดตรวจสอบสิทธิ์" โดยแบ่งเป็น 3 ส่วนหลัก:

ส่วนที่ 1: การอัปเดตระบบหลังบ้าน (Supabase SQL)
เราจะแก้ไข SQL ฟังก์ชันส่วนกลาง เพื่อให้ระบบเดิมที่เคยตรวจสอบผ่าน is_admin() หรือนโยบาย RLS เข้าใจว่า SUPER_ADMIN ก็คือผู้ดูแลระบบเช่นกัน

ปรับปรุง Helper Function ในการตรวจเช็คสิทธิ์ ( SQL Editor ):
sql
-- 1. ปรับฟังก์ชันตรวจสอบว่าเป็น Admin หรือ Superadmin (สิทธิ์ระดับแอดมินทั้งหมด)
CREATE OR REPLACE FUNCTION public.is_admin()
RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.users WHERE id = auth.uid() AND role IN ('ADMIN', 'SUPER_ADMIN')
  )
$$ LANGUAGE sql SECURITY DEFINER STABLE;
-- 2. ปรับฟังก์ชันระบุตัวตนของ Superadmin โดยเฉพาะ (สำหรับสิทธิ์สูงสุด)
CREATE OR REPLACE FUNCTION public.is_super_admin()
RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.users WHERE id = auth.uid() AND role = 'SUPER_ADMIN'
  )
$$ LANGUAGE sql SECURITY DEFINER STABLE;
ปรับปรุงสิทธิ์ RLS บนตาราง audit_logs และ academic_years (จำกัดเฉพาะ Superadmin เท่านั้น):
sql
-- ตัวอย่าง: ตาราง audit_logs ให้เฉพาะ Superadmin เข้าถึงได้
DROP POLICY IF EXISTS "admin_select" ON public.audit_logs;
CREATE POLICY "super_admin_select"
  ON public.audit_logs FOR SELECT
  TO authenticated
  USING (public.is_super_admin());
ส่วนที่ 2: การปรับปรุงโค้ดตรวจสอบสิทธิ์หน้าบ้าน (Frontend Auth)
เราจะสร้างตัวช่วยที่ง่ายขึ้นในระบบ React เพื่อประเมินสิทธิ์ของผู้ใช้

อัปเดต 

authStore.ts
 เพื่อให้ง่ายต่อการดึงค่า:
typescript
// เพิ่ม helper getters
export const useAuth = () => {
  const { user, role, signOut } = useAuthStore();
  return {
    user,
    role,
    signOut,
    isSuperAdmin: role === 'SUPER_ADMIN',
    isAdmin: role === 'ADMIN' || role === 'SUPER_ADMIN', // ครอบคลุมทั้งสองตำแหน่ง
    isTeacher: role === 'TEACHER' || role === 'ADVISOR',
    isParent: role === 'PARENT',
  };
};
การปรับปรุงการเข้าหน้าหลักใน 

App.tsx
:
typescript
const RootDashboard = () => {
  const { role } = useAuthStore()
  // ให้ Superadmin และ Admin ทั่วไปเข้าใช้งานหน้า Dashboard เดียวกันได้ทันที โค้ดส่วนนี้จะไม่พัง
  if (role === 'ADMIN' || role === 'SUPER_ADMIN') return <Dashboard />
  if (role === 'TEACHER' || role === 'ADVISOR') return <TeacherDashboard />
  ...
}
ส่วนที่ 3: ปิดฟีเจอร์ระดับสูงสำหรับ Admin ทั่วไป (UI Restrictions)
ในหน้า Admin Dashboard และตั้งค่าระบบ เราสามารถใช้เงื่อนไขตรวจสอบเพื่อ ซ่อน/แสดง ปุ่มหรือแถบเมนูตามระดับความลึกของสิทธิ์ได้ เช่น:

การจัดการปีการศึกษา (AcademicYears.tsx): ปุ่ม "+ เพิ่มปีการศึกษา" หรือปุ่มบันทึกแก้ไข จะซ่อนไว้หากไม่ใช่ Superadmin
tsx
const { role } = useAuthStore();
const isSuperAdmin = role === 'SUPER_ADMIN';
// ในส่วน JSX:
{isSuperAdmin && (
  <button className="bg-indigo-600 ...">
    + เพิ่มปีการศึกษา
  </button>
)}
ระบบแต่งตั้งผู้ใช้งาน (Settings.tsx หรือหน้า User Management): แอดมินปกติทำได้แค่อ่านข้อมูล แต่ Superadmin เท่านั้นที่จะเห็นปุ่มหรือ Dropdown สำหรับอัปเดตสิทธิ์ผู้ใช้
💡 ข้อดีของแนวทางนี้:
โค้ดเดิมไม่พัง: เนื่องจากเรากำหนดให้ isAdmin = true เมื่อผู้ใช้เป็นทั้ง ADMIN และ SUPER_ADMIN ส่งผลให้เงื่อนไขเช็คสิทธิ์เดิมในระบบยังทำงานต่อได้ตามปกติโดยไม่มีบั๊กเกิดขึ้น
ทำได้ไว: ปรับเปลี่ยนเพียงไม่กี่จุดก็พร้อมใช้
ปลอดภัย: ควบคุมสิทธิ์จากทั้งฝั่ง PostgreSQL (ฐานข้อมูล) และ React (หน้าบ้าน)
หากคุณครูมีความคิดเห็นเพิ่มเติมหรือต้องการให้ผมร่างแผนงานสำหรับการปรับเปลี่ยนตัวสิทธิ์นี้เพื่อลงมือเขียนโค้ดต่อ สามารถแจ้งผมได้เลยนะครับ!
