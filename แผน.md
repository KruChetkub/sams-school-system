# แผนการเปิดใช้งาน RLS และตรวจสอบความปลอดภัยแยกตามตาราง (Database RLS & Security Verification Plan)

แผนงานนี้จัดทำขึ้นเพื่อใช้ในการทยอยเปิดใช้งาน **Row Level Security (RLS)** ในฐานข้อมูล และทำการทดสอบสิทธิ์การเข้าถึงข้อมูลทีละตารางอย่างละเอียด เพื่อแก้ไขช่องโหว่ความปลอดภัยที่ตรวจพบใน `Security Advisor.md` ให้เสร็จสิ้นสมบูรณ์

---

## 🛠️ ขั้นตอนมาตรฐานในการจัดการแต่ละตาราง (Standard Operating Procedure)

สำหรับแต่ละตารางที่ทำดำเนินการเปิดระบบความปลอดภัย จะต้องผ่านกระบวนการ 4 ขั้นตอนดังนี้:

1. **Enable RLS**: รันคำสั่ง SQL เพื่อเปิดใช้งาน RLS บนตาราง
   ```sql
   ALTER TABLE public.ตาราง ENABLE ROW LEVEL SECURITY;
   ```
2. **Apply Policies**: เขียนและอัปเดตนโยบาย RLS (SELECT, INSERT, UPDATE, DELETE) ให้รองรับสิทธิ์ของแต่ละกลุ่มผู้ใช้งาน (รวมถึงสิทธิ์ของครูที่ปรึกษาคนที่ 1 และคนที่ 2)
3. **Verify Security (Direct API Test)**: ทดลองเรียกใช้งาน REST API (PostgREST) ดึงข้อมูลโดยไม่ระบุสิทธิ์/ส่งคีย์ผู้ใช้ทั่วไป (Anonymous Request) -> ต้องไม่สามารถเห็นข้อมูล (ได้ค่าว่าง `[]` หรือ `401 Unauthorized`)
4. **App Integration Test (Frontend Test)**: เข้าสู่ระบบผ่านหน้าเว็บจริงด้วยบทบาทต่าง ๆ (Super Admin, Admin, ครูที่ปรึกษาห้องนั้น, ครูประจำวิชาอื่น, ผู้ปกครอง) เพื่อทดสอบการทำงานว่าลื่นไหลและปลอดภัยจริง

* **นโยบาย**: เปิดให้ครูประจำชั้น (คนที่ 1 และคนที่ 2) ของนักเรียนเข้าถึงบันทึกเยี่ยมบ้าน อัปรูปภาพ และแก้ไขประเมินความเสี่ยงได้ / แอดมินและซูเปอร์แอดมินจัดการได้ครอบคลุม
- [ ] เปิดใช้งาน RLS (`ALTER TABLE ENABLE RLS` ทุกตารางในกลุ่มเยี่ยมบ้าน)
- [ ] เขียนนโยบาย RLS (สิทธิ์ผูกโยงระหว่างครูประจำชั้นคนที่ 1 และ 2)
- [ ] ทดสอบความปลอดภัย:
  - [ ] Anonymous Bypass: บล็อกการคิวรีหรืออัปโหลดรูปภาพเยี่ยมบ้านโดยไม่ล็อกอิน
  - [ ] Co-advisor Access: ครูที่ปรึกษาคนที่ 1 บันทึกเยี่ยมบ้าน -> ครูคนที่ 2 สามารถเปิดดูรูปภาพ แผนที่ พิมพ์รายงาน PDF สรุปเยี่ยมบ้านของเด็กคนนั้นที่มีการลงลายชื่อครูทั้งสองท่านได้สมบูรณ์
  - [ ] Data Confidentiality: ป้องกันไม่ให้ครูห้องอื่นเห็นข้อมูลเยี่ยมบ้าน/รูปภาพและแผนที่พิกัดบ้านของเด็กห้องนี้
- [ ] สรุปผลการทดสอบ: ปลอดภัยและเข้าใช้งานได้ถูกต้อง

#### 14. ตารางคัดกรองช่วยเหลือ (`student_support_sdq`, `student_support_eq`, `student_support_risk_analysis`)

- [ ] เปิดใช้งาน RLS (`ALTER TABLE ENABLE RLS` สำหรับทุกตารางในระบบดูแลช่วยเหลือ)
- [ ] เขียนนโยบาย RLS (จำกัดสิทธิ์อ่านเขียนเฉพาะครูประจำชั้น 1 & 2 และผู้ประเมิน)
- [ ] ทดสอบความปลอดภัย:
  - [ ] Security Leak Test: ครูประจำชั้นห้องอื่นทดลองคิวรีหรือเข้าหน้าพฤติกรรมเด็กห้องอื่น -> ระบบต้องตอบกลับปฏิเสธสิทธิ์หรือกรองไม่พบ
  - [ ] Real-time Evaluation: ครูประจำชั้นคนที่ 1 กรอกผลคัดกรอง -> ครูคนที่ 2 และแอดมินมองเห็นระดับความเสี่ยงของนักเรียนอัปเดตแบบเรียลไทม์ทันทีในแดชบอร์ด
- [ ] สรุปผลการทดสอบ: ปลอดภัยและเข้าใช้งานได้ถูกต้อง

### 👁️ กลุ่มที่ 6: การปรับปรุง SQL View ที่มีความเสี่ยงด้านความปลอดภัย

#### 16. ปรับเปลี่ยนคุณสมบัติของวิว `public.active_students`
* **ประเด็น**: วิวเดิมใช้ `SECURITY DEFINER` ซึ่งรันคำสั่งโดยใช้สิทธิ์ของระบบ/แอดมินผู้สร้างวิว ทำให้ผู้ใช้คนใดก็ตามที่คิวรีผ่านวิวนี้จะข้ามกฎ RLS ทั้งหมดไปโดยปริยาย
* **แนวทางแก้ไข**: ปรับวิวใหม่ให้ใช้ `SECURITY INVOKER` เพื่อให้กฎ RLS ของผู้ที่ส่งคำขอนั้นถูกนำมาบังคับใช้อย่างสมบูรณ์
- [ ] รัน SQL เพื่อดรอปวิวและสร้างใหม่ด้วยคุณสมบัติ `SECURITY INVOKER`
- [ ] ทดสอบความปลอดภัย:
  - [ ] รันคำสั่ง SQL Query บนบอร์ดทดลองผ่านสิทธิ์ครูแต่ละห้อง ดึงข้อมูลผ่านวิว `public.active_students`
  - [ ] ตรวจสอบผลลัพธ์ว่าผลที่แสดงออกมาโดน RLS กรองให้เห็นเฉพาะเด็กที่ครูท่านนั้นมีสิทธิ์ดูแลเท่านั้น (ไม่ได้แสดงรายชื่อเด็กทุกคนในโรงเรียนเหมือนแต่ก่อน)
  - [ ] ตรวจสอบว่าใน Supabase Security Panel การแจ้งเตือนช่องโหว่ความเสี่ยงระดับ ERROR ของวิวนี้ได้รับการแก้ไขและหายไปเรียบร้อยแล้ว
- [ ] สรุปผลการทดสอบ: ปลอดภัยและเข้าใช้งานได้ถูกต้อง
