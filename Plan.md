# แผนการพัฒนาระบบ Portal และระบบเยี่ยมบ้านนักเรียน (Home Visit System)

เอกสารนี้อธิบายถึงสถาปัตยกรรม แผนการออกแบบ และขั้นตอนการพัฒนา เพื่อรองรับ **ระบบหลายแอปพลิเคชัน (Multi-App System)** โดยมีจุดศูนย์กลาง (Portal) สำหรับเลือกเข้าใช้งานระบบต่างๆ และแผนการพัฒนา **ระบบเยี่ยมบ้านนักเรียน (Home Visit System)** โดยจะใช้ฐานข้อมูลนักเรียนร่วมกับระบบเช็คชื่อ (SAMS) เดิม

---

## 1. สถาปัตยกรรมระบบใหม่ (New System Architecture)

เพื่อให้ระบบรองรับการขยายตัวในอนาคต (เช่น เพิ่มระบบห้องสมุด, ระบบพฤติกรรม) โดยไม่ให้โค้ดส่วน Frontend ปะปนกันจนซับซ้อนเกินไป เราจะปรับสถาปัตยกรรมดังนี้:

### 1.1 App Portal (หน้าเลือกแอปพลิเคชัน)
- **Flow เดิม:** Login -> เข้าสู่หน้า Dashboard ของ SAMS ทันที
- **Flow ใหม่:** Login -> เข้าสู่หน้า **App Portal (`/portal`)** -> แสดงการ์ดระบบที่ผู้ใช้มีสิทธิ์เข้าถึง -> คลิกเลือกแอป -> เข้าสู่แอปนั้นๆ

### 1.2 Routing Namespace (การจัดการ URL)
เพื่อแยกแยะว่าผู้ใช้กำลังอยู่ในระบบใด จะมีการปรับโครงสร้าง Route ของ React Router:
- `/portal`: หน้าเลือกแอปพลิเคชัน
- `/sams/*`: (หรือคงไว้ที่ Root `/` ขึ้นอยู่กับข้อตกลง) สำหรับระบบ SAMS เดิม เช่น `/sams/dashboard`, `/sams/schedules`
- `/homevisit/*`: สำหรับระบบเยี่ยมบ้าน เช่น `/homevisit/dashboard`, `/homevisit/students`

### 1.3 State Management & Layouts
- **แยก Sidebar / Navbar:** แต่ละระบบจะมี Sidebar ของตัวเอง (SAMS มีเมนูเช็คชื่อ, Home Visit มีเมนูบันทึกการเยี่ยมบ้าน)
- **Shared Context:** ใช้ `AuthContext` ร่วมกัน และอาจมี `AppCtx` (App Context) เพื่อบอกว่าตอนนี้ผู้ใช้กำลัง Active อยู่ที่แอปไหน เพื่อให้ UI/Theme ปรับเปลี่ยนตามแอปนั้นๆ

### 1.4 โครงสร้างโฟลเดอร์ (Directory Structure & Code Isolation)
เพื่อรับประกันว่า **การพัฒนาระบบเยี่ยมบ้านจะไม่ทำให้ระบบ SAMS เดิมพัง** และโค้ดเดิมยังคงทำงานได้ปกติ 100% ในช่วงพัฒนา เราจะใช้โครงสร้างแบบ **Sub-module Isolation**:
- **ไฟล์ของ SAMS (ระบบเดิม):** ปล่อยไว้ที่เดิมทั้งหมด (เช่น `src/pages/Dashboard.tsx`) จะไม่มีการเปลี่ยนชื่อหรือย้ายโฟลเดอร์ใดๆ เพื่อป้องกันบัค
- **ไฟล์ของ Home Visit (ระบบใหม่):** จะสร้างโฟลเดอร์ย่อยชื่อ `homevisit` ในแต่ละส่วนแยกออกมาอย่างชัดเจน
- **กฎเหล็ก (Strict Rule):** ห้ามไฟล์ในระบบ Home Visit ดึงโค้ด (Import) จากหน้าจอของ SAMS มาใช้เด็ดขาด ถ้ามีฟังก์ชันที่ต้องใช้ซ้ำกัน ให้แยกโค้ดนั้นออกมาไว้ในโฟลเดอร์ `shared` หรือโฟลเดอร์กลางเท่านั้น

**ตัวอย่างโครงสร้างที่จะเพิ่มเข้าไป:**
```text
src/
 ├── pages/
 │    ├── ... (ไฟล์ระบบ SAMS เดิม ปล่อยไว้ที่เดิมทั้งหมด)
 │    ├── portal/           <-- (ใหม่) หน้า App Portal สำหรับเลือกระบบตอนล็อกอิน
 │    └── homevisit/        <-- (ใหม่) หน้าจอทั้งหมดของระบบเยี่ยมบ้านแยกไว้ที่นี่
 │         ├── Dashboard.tsx
 │         └── Students.tsx
 │
 ├── services/
 │    ├── ... (ไฟล์ API เดิม ปล่อยไว้ที่เดิม)
 │    └── homevisit/        <-- (ใหม่) API สำหรับเรียกข้อมูลเยี่ยมบ้านโดยเฉพาะ
 │
 └── components/
      ├── ... (ไฟล์ UI เดิม ปล่อยไว้ที่เดิม)
      └── homevisit/        <-- (ใหม่) Component ย่อยที่ใช้เฉพาะในระบบเยี่ยมบ้าน
```

## 2. โครงสร้างฐานข้อมูล (Database Design)

ระบบใหม่จะพึ่งพาข้อมูลหลัก (Master Data) จากระบบ SAMS เพื่อลดความซ้ำซ้อน

### 2.1 ข้อมูลที่ใช้งานร่วมกัน (Shared Data - มีอยู่แล้ว)
- `users`: ข้อมูลครูผู้สอน/บุคลากร
- `students`: ข้อมูลนักเรียนพื้นฐาน
- `classrooms`: ข้อมูลห้องเรียน
- `student_classrooms`: การผูกนักเรียนเข้ากับห้องเรียนในแต่ละปีการศึกษา

### 2.2 ข้อมูลใหม่สำหรับระบบเยี่ยมบ้าน (New Tables สำหรับ Home Visit)
- **`home_visits` (ข้อมูลการเยี่ยมบ้าน)**
  - `id` (PK)
  - `student_id` (FK -> students)
  - `teacher_id` (FK -> users)
  - `visit_date` (วันที่ไปเยี่ยม)
  - `status` (สถานะ: นัดหมายแล้ว, เยี่ยมแล้ว, ยกเลิก)
  - `latitude`, `longitude` (พิกัด GPS ของบ้านนักเรียน)
  - `address_details` (รายละเอียดที่อยู่เพิ่มเติม)
  - `created_at`, `updated_at`

- **`home_visit_assessments` (แบบประเมิน/บันทึกผลการเยี่ยมบ้าน)**
  - `id` (PK)
  - `visit_id` (FK -> home_visits)
  - `living_condition` (สภาพความเป็นอยู่)
  - `family_status` (สถานภาพครอบครัว)
  - `economic_status` (ฐานะทางเศรษฐกิจ)
  - `risk_level` (ระดับความเสี่ยง: ปกติ, เฝ้าระวัง, ช่วยเหลือเร่งด่วน)
  - `notes` (บันทึกเพิ่มเติม)

- **`home_visit_photos` (รูปภาพประกอบ)**
  - `id` (PK)
  - `visit_id` (FK -> home_visits)
  - `photo_url` (URL ของรูปภาพใน Storage)
  - `description` (คำอธิบายรูป)

## 3. ฟีเจอร์หลักของระบบเยี่ยมบ้าน (Home Visit Features)

1. **Dashboard การเยี่ยมบ้าน (Executive/Teacher Dashboard)**
   - สรุปความคืบหน้าการเยี่ยมบ้าน (คิดเป็นเปอร์เซ็นต์ของนักเรียนที่รับผิดชอบ)
   - แผนภูมิแสดงระดับความเสี่ยงของนักเรียน (ปกติ/เสี่ยง/เร่งด่วน)

2. **การจัดการแผนการเยี่ยมบ้าน (Visit Planner)**
   - ดูรายชื่อนักเรียนในที่ปรึกษา
   - กำหนดวันที่และนัดหมายการเยี่ยมบ้าน
   - แสดงสถานะ (ยังไม่เยี่ยม, นัดหมายแล้ว, เรียบร้อยแล้ว)

3. **ฟอร์มบันทึกการเยี่ยมบ้าน (Visit Assessment Form)**
   - ดึงข้อมูลพื้นฐานนักเรียนมาแสดงอัตโนมัติ
   - ฟอร์มกรอกข้อมูลสภาพแวดล้อม ครอบครัว และเศรษฐกิจ
   - ระบบแนบรูปถ่ายสภาพบ้าน
   - ระบบปักหมุด GPS (ดึง Location จากอุปกรณ์พกพา)

4. **ระบบรายงาน (Reporting)**
   - พิมพ์แบบฟอร์มรายงานการเยี่ยมบ้าน (PDF) รายบุคคล
   - พิมพ์รายงานสรุปภาพรวมรายห้องเรียน

## 4. ขั้นตอนการดำเนินการ (Implementation Phases)

### Phase 1: Foundation & Portal (ปรับโครงสร้างเป็น Multi-App)
1. สร้างหน้า `/portal` สำหรับเป็นจุดเชื่อมต่อ (Hub)
2. ปรับปรุง React Router โยกย้าย Routes เดิมของ SAMS ไปไว้ใน Namespace ของมันเอง (หรือปล่อยไว้ แต่เตรียมรองรับ Router ของ Home Visit)
3. ปรับโครงสร้าง Layout/Sidebar ให้รองรับการสลับแอป (Switch App) โดยที่ Sidebar เมนูเปลี่ยนไปตามระบบที่เลือก
4. **เป้าหมาย Phase 1:** ผู้ใช้ Login เข้ามาเจอหน้าการ์ดให้เลือกระบบ กด SAMS แล้วทำงานได้เหมือนเดิมทุกประการ

### Phase 2: Home Visit Database & Backend
1. สร้างตาราง `home_visits`, `home_visit_assessments`, `home_visit_photos` ใน Supabase
2. ตั้งค่า RLS (Row Level Security) สำหรับตารางใหม่ ให้ครูเห็นเฉพาะข้อมูลนักเรียนที่ตนเองรับผิดชอบ
3. สร้าง Storage Bucket สำหรับเก็บรูปภาพการเยี่ยมบ้าน

### Phase 3: Home Visit Frontend (UI & Features)
1. พัฒนาหน้า Dashboard สำหรับ Home Visit
2. พัฒนาหน้าหน้ารายชื่อนักเรียน (ดึงจากตาราง `students` และ `student_classrooms`) และสถานะการเยี่ยมบ้าน
3. พัฒนาฟอร์มบันทึกการเยี่ยมบ้าน (กรอกข้อมูล, อัปโหลดรูป, บันทึก GPS)
4. ปรับ Theme ของระบบเยี่ยมบ้านให้มีเอกลักษณ์ (อาจจะใช้โทนสีต่างจาก SAMS เล็กน้อยเพื่อให้ผู้ใช้รู้ตัวว่าอยู่คนละระบบ)
5. **การตั้งค่าครูที่ปรึกษา (Classroom Advisor Management)**: เพิ่มหน้าจอ/ระบบสำหรับ Admin ในการจับคู่ว่าครูท่านใดเป็นครูประจำชั้นของห้องไหน (`advisor_id` ใน `classrooms`) เพื่อให้ระบบ Role-based access สามารถแสดงเด็กได้ตรงกับครูที่ปรึกษา

### Phase 4: Reporting & Export
1. เพิ่มฟังก์ชัน Export รายงานการเยี่ยมบ้านเป็น PDF
2. นำข้อมูลระดับความเสี่ยงที่ได้จากการเยี่ยมบ้าน ไปต่อยอดหรือแสดงไอคอนแจ้งเตือนในระบบ SAMS (Cross-system integration)

## 5. ความท้าทายและการเตรียมการ
- **Shared Data Integrity:** ต้องระวังการลบข้อมูลนักเรียนใน SAMS จะกระทบประวัติการเยี่ยมบ้าน ควรใช้ Soft Delete หรือ RLS ที่รัดกุม
- **Mobile First:** ระบบเยี่ยมบ้าน ครูมักจะใช้งานบนมือถือหรือแท็บเล็ตขณะลงพื้นที่ ดังนั้น UI ต้องเป็น Mobile-Responsive 100%
- **Offline Capabilities:** (อาจพิจารณาในอนาคต) การลงพื้นที่บางที่อาจไม่มีอินเทอร์เน็ต อาจต้องมีระบบเซฟฟอร์มไว้ในเครื่อง (Local Storage) แล้วค่อย Sync กลับเมื่อมีเน็ต
